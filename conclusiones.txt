CONCLUSIONES Y HALLAZGOS – serenity-demoblaze (Serenity BDD + Cucumber + SerenityRest)

HALLAZGOS
1) Demoblaze presenta comportamiento no determinista en UI:
   - Los alerts del navegador al agregar al carrito pueden aparecer con latencia.
   - El listado/catálogo de productos puede variar entre ejecuciones.
   Mitigación aplicada:
   - Se implementó aceptación de alert con reintento/espera explícita.
   - Se optó por una selección de productos estable para reducir flakiness.

2) Las APIs de Demoblaze responden HTTP 200 tanto para éxito como para error.
   Por lo tanto, el criterio correcto de validación es el contenido del body:
   - Éxito en signup puede devolver body vacío ("").
   - Error devuelve JSON con "errorMessage".
   - Login exitoso devuelve string con "Auth_token: ...".
   Mitigación aplicada:
   - Validación por contenido del body (texto crudo) y verificación de presencia/ausencia
   de "errorMessage" y/o "Auth_token".

3) Riesgo de inconsistencias de encoding (tildes) en steps escritos en archivos .feature.
   Mitigación aplicada:
   - Se estandarizaron steps sin caracteres especiales y/o se aseguró encoding UTF-8 para
   evitar fallos por interpretación de caracteres.
